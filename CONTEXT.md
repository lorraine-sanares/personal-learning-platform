# Domain Glossary

## Insight
The atomic unit of knowledge in this system. A distilled idea or understanding extracted from one or more Sources. An Insight is authored by the system (via synthesis) or by the user directly. It is the thing that gets stored, reviewed, and connected to other Insights. It links back to every Source it was derived from.

An Insight has one of two states:
- **Committed** — AI-extracted with high confidence, immediately part of the knowledge base
- **Draft** — flagged as ambiguous (e.g. from a voice memo, image, or unstructured note); sits in the Backlog until the user maps it to a Theme

## Backlog
The queue of Draft Insights awaiting the user's review. The user's job in the Backlog is to assign each Draft Insight to one or more Themes, after which the Insight becomes Committed.

## Theme
An overarching topic that organises Insights. Themes follow a hybrid model: the user seeds an initial set of broad Themes manually, and the AI proposes new granular Themes when enough Insights cluster around a topic that has no existing home. The user approves or renames each proposed Theme before it is committed. Themes are nodes in the Knowledge Graph and connect to each other where overlap exists.

## Card
A question-answer pair generated by the AI from a single Insight. Cards are the unit of spaced repetition review — the user answers the question before the answer is revealed, then self-rates difficulty (Again / Hard / Good / Easy). A Card always traces back to exactly one Insight. One Insight can produce multiple Cards. A Card cannot exist without a parent Insight.

Scheduling uses the **FSRS algorithm**: each Card tracks `stability`, `difficulty`, and `due_date`. The system selects Cards for a Review Session via the Review Queue.

## Review Queue
The ordered list of Cards selected for a given Review Session. Bounded by a configurable **Daily Card Limit** (a small, fixed cap — e.g. 10–20 Cards) so review load never accumulates into an overwhelming backlog. Within the cap, Cards are prioritised by FSRS retention risk — Cards closest to being forgotten appear first.

## Knowledge Graph
The network of Themes and their relationships, built up over time as Insights are mapped to Themes and Themes are linked to one another. Visualised as a graph (similar to Obsidian's graph view) to give the user a bird's-eye view of their overall knowledge base.

## Explore View
A dedicated app view for free-form navigation of the user's personal knowledge base. The user browses Themes, traverses the Knowledge Graph, and discovers connections between Insights — without a fixed agenda. Explore is entirely scoped to the user's own vault; it does not search the web.

## Insights Dashboard
The main consumption hub of the app. An observability-style interface presenting the knowledge base through four Consumption Modes the user can switch between:

- **Snack** — a single Insight or short summary for quick consumption (commute, spare minute)
- **Recent** — a reverse-chronological feed of newly committed Insights
- **Digest** — a browsable archive of past morning Digests
- **Deep Dive** — a full-context view of a single Insight: all its parent Themes, related Insights, and direct Source links (URLs, file references)

## Consumption Mode
The lens through which the user consumes content in the Insights Dashboard. One of: Snack, Recent, Digest, Deep Dive.

## Perspective
A first-person opinion or stance authored by the user in response to a Reflection Prompt. Unlike an Insight (which is AI-extracted and factual), a Perspective is subjective and represents the user's evolving point of view. Perspectives are linked to one or more Themes and Insights from the knowledge base, allowing the system to surface the Themes around which the user's opinions are forming. Perspectives are displayed in the Perspectives Tab.

## Reflection Prompt
A question posed by the agent to solicit a Perspective from the user — e.g. "What are your thoughts on [Topic]?" Reflection Prompts are generated proactively by the agent, based on the Themes the user has been engaging with most and Insights from recent Digests. This is the primary surface for topic recommendations — the agent uses engagement history to decide what to ask next. The user may respond immediately or defer.

## Perspectives Tab
A dedicated app view displaying all of the user's authored Perspectives, organised by Theme. Shows the evolution of the user's thinking over time — which Themes their opinions cluster around, and how their stance on a topic has shifted as their knowledge base grew. Each Perspective links directly to the Insights and Sources that informed it.

## Knowledge Context Panel
An on-demand panel available while reading any Insight. Collapsed by default to preserve the clean reading surface. When opened, it shows: the Themes this Insight belongs to, semantically related Insights, and a mini Knowledge Graph centred on this Insight. Closed again with one click.

## Digest
A pre-generated morning document containing: (1) Insights extracted from Sources ingested the previous day, and (2) Related Content — web and news articles the system discovers that are thematically relevant to those Sources. The Digest is generated on a daily schedule and delivered in two forms: as an EPUB exported to the user's Kindle (V1), and as a readable view inside the app (V1). It is a passive reading experience — no interaction required.

Visual format: a stats header ("X Insights across Y Themes"), each Insight rendered as a styled summary card (headline + 2-line summary + Theme tag), and og:image thumbnails pulled from article Sources.

## Related Content
Web and news articles that the system proactively discovers and surfaces in the Digest because they are thematically relevant to recently ingested Sources. Related Content is AI-sourced (not manually ingested) and is read-only — it enriches the Digest but does not automatically generate Insights unless the user explicitly ingests the article as a Source.

## Review Session
An interactive evening session inside the app. It presents spaced repetition Cards drawn from two pools: (1) Cards from previous days that are due for review per the spaced repetition schedule, and (2) Cards generated from Insights that appeared in that same day's Digest. The user answers each Card, then self-rates difficulty, which updates the Card's next review date.

## Tech Stack (V1)
- **Frontend + API** — Next.js (full-stack, handles UI and API routes in one project)
- **Database** — SQLite via Drizzle ORM (single-user, no server setup; upgradeable to Postgres for V2)
- **AI: synthesis, Digest, Cards** — Claude Sonnet 4.6 (Insight extraction, Card generation, Digest authoring)
- **AI: fast classification** — Claude Haiku 4.5 (Theme classification, OCR structuring)
- **PDF + image parsing** — pdf-parse + Tesseract.js (server-side in Next.js)
- **EPUB generation** — epub-gen (Node-native, no system dependencies)
- **Spaced repetition** — ts-fsrs (TypeScript FSRS implementation)

## Platform Roadmap
- **V1** — responsive web app, desktop browser + EPUB export to Kindle
- **V2** — native desktop app (Mac)
- **V3** — native iOS app

## Obsidian Vault
A sync target — a read-friendly downstream copy of the knowledge base written as `.md` files. The app's database is the authoritative source of truth. After each write (new Insight, Theme, Card), the system exports a corresponding `.md` file to the vault. Edits made inside Obsidian do not flow back to the app (bi-directional sync is a future feature).

## Source
A raw input from which Insights are extracted. A Source has not yet been synthesized — it is the unprocessed material that feeds the system. Every Insight traces back to at least one Source.

Supported Source types (v1 — all manually ingested):
- **PDF** — a full uploaded document
- **Article** — a URL pasted by the user; content is fetched and extracted
- **Meeting Note** — text imported from Apple Notes or OneNote
- **Workspace Page** — a page from Notion or Obsidian
- **Image** — a photo of handwriting or printed text; transcribed via OCR before processing

Future: passive ingestion via email forwarding, calendar sync, and mobile capture (logged for v2+).
